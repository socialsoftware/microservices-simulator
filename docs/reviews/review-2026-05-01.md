**Additional patterns from Check 8.7 (EventProcessing design):**

Pattern 7 — EventProcessing design divergence:

`docs/concepts/events.md` lines 170–184 (canonical wiring):
```java
@Service
public class <Consumer>EventProcessing {
    @Autowired
    private <Consumer>Functionalities <consumer>Functionalities;

    public void process<Xxx>Event(Integer aggregateId, <EventName> event) {
        <consumer>Functionalities.<updateMethod>(aggregateId, event.get<RelevantField>());
    }
}
```
Method signature: `process{Xxx}Event(Integer aggregateId, {EventName} event)`. Delegates to Functionalities, which opens its own UoW.

`session-d.md` lines 70–79:
```
- One `process{Event}(Saga{Aggregate} aggregate, {Event} event)` method per subscribed event
- Each method:
  1. Check aggregate.getSagaState() != NOT_IN_SAGA
  2. Update cached snapshot fields directly on aggregate
  3. For deletion events: mark aggregate/sub-entity deleted/invalid
  4. Call aggregate.verifyInvariants()
  5. Persist the updated aggregate
```
Method signature: `process{Event}(Saga{Aggregate} aggregate, {Event} event)`. Directly mutates aggregate, no Functionalities delegation.

`docs/architecture.md` lines 43–47 and `docs/concepts/rule-enforcement-patterns.md` lines 207–213 both show the events.md design (EventProcessing delegates to Functionalities). **Critical** — an agent implementing session-d produces an EventProcessing that contradicts the canonical wiring in events.md and the architecture description.
