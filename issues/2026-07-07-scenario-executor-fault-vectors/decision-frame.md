## Repo Context

- `ScenarioPlan` already carries `FaultSpace` with fault slot count, scheduled step ids, and all-zero default vector; individual fault vectors are not separate scenario identities.
- The current verifier-owned `ScenarioExecutor` can execute only supported single-saga plans, materialize selected inputs, and call scheduled steps via `executeUntilStep(...)`.
- Existing simulator fault behavior is implemented in `ExecutionPlan` through `ImpairmentHandler`, currently loading per-functionality CSV files into `stepName -> [fault, delayBefore, delayAfter]` maps.
- Quizzes fault tests reuse simulator impairment through behavior directories/CSV files, but that path is application-test-resource oriented and not a good canonical executor contract.
- `Workflow.execute(...)` and `Workflow.resume(...)` own commit/abort/compensation semantics; a fault thrown from raw `executeUntilStep(...)` can bypass the normal abort/compensation path.
- Current docs explicitly list generated fault injection, behavior generation, impact scoring, GA search, and prioritization as not implemented.

## Problem Framing

The verifier needs scenario execution attempts to run with an assigned binary fault vector over catalog fault slots, while preserving simulator-owned fault injection and saga lifecycle semantics. This should turn the executor from fault-free schedule replay into a stable execution stage that future impact scoring, local search, and scenario prioritization can consume, without making CSV files or Quizzes-specific tests the core contract.

## Selected Direction

Introduce assigned binary fault vectors as executor input and route them through a simulator-side in-memory fault injection provider. The executor maps a vector onto `ScenarioPlan.faultSpace` slots, installs a scenario-scoped provider, executes the workflow through lifecycle paths that preserve abort/compensation behavior, and records the effective vector plus slot mapping in the execution report. Existing CSV behavior remains a compatibility provider for current tests/manual scripts, not the canonical executor path.

## Alternatives Considered

| Option | Status | Summary | Reason |
|--------|--------|---------|--------|
| In-memory simulator fault provider | selected | Add a simulator abstraction queried by `ExecutionPlan`; executor installs a scenario-scoped provider backed by the assigned fault vector. | Removes CSV from the executor contract while keeping simulator-owned fault injection and preserving a path to multi-saga/search later. |
| CSV adapter generated by executor | rejected | Executor writes per-functionality behavior CSV files from the assigned vector and points `ImpairmentHandler` at them. | Fastest compatibility bridge, but keeps weak file/global-state/simple-name semantics as core executor behavior. |
| Minimal in-memory registry inside `ImpairmentHandler` only | deferred | Let `ImpairmentHandler.loadStepsFile(...)` return an active in-memory assignment before falling back to CSV. | Useful transitional implementation if needed, but still centers the old handler/global-state model. |
| Generated test/facade-style harness | rejected | Generate test-like code/resources that mimic Quizzes fault tests. | Too application/test-lifecycle coupled; weak fit for a generic scenario executor. |

## Key Tradeoffs

| Tradeoff | Chosen Side | Consequence |
|----------|-------------|-------------|
| CSV compatibility vs executor contract quality | In-memory provider as canonical contract | Requires simulator refactor, but avoids file-format debt in the executor. |
| Executor-owned fault throwing vs simulator-owned injection | Simulator-owned injection | Fault behavior stays aligned with workflow execution and dynamic evidence hooks. |
| Binary simplicity vs broader impairment model | Binary-only vectors | Keeps first fault contract deterministic and search-friendly; delays/fault types remain separate future work. |
| Short implementation path vs production-shaped lifecycle | Preserve workflow lifecycle semantics | More executor orchestration work, but avoids invalid compensation/abort observations. |
| Current single-saga support vs future multi-saga design | Design for exact scenario/saga/step identity | First runnable scope may still be single-saga, but APIs should not depend only on simple functionality names or CSV counters. |

## Assumptions

- Fault vector bits are strictly binary: `0` means no injected fault, `1` means simulator exception at that scheduled step.
- Fault vectors are assigned to an execution; the executor should not randomly generate them in this feature.
- `ScenarioPlan` identity remains independent from fault vector identity.
- CSV behavior remains supported for existing tests unless explicitly removed in a later compatibility decision.
- The executor may implement single-saga execution first, but the spec should describe the target identity model needed for multi-saga and search.

## Non-Goals

- Delay injection or non-binary impairment types.
- Genetic/local search implementation.
- Domain-impact scoring implementation.
- Scenario prioritization or contextual bandit implementation.
- Quizzes-specific executor shortcuts.
- Treating generated CSV files as the long-term executor contract.

## Open Questions for Spec

| ID | Question | Why It Matters | Risk | Recommended Default |
|----|----------|----------------|------|---------------------|
| Q1 | Should the executable acceptance scope include multi-saga faulted scenarios now, or should the spec require production-shaped identity/provider APIs while accepting only single-saga execution initially? | Multi-saga runtime scheduling materially changes executor behavior, state setup, and validation cost. | high | Implement single-saga fault execution now, but require provider/query/report models that carry scenario plan id, saga instance id, scheduled step id, and runtime step name so multi-saga is not blocked. |
| Q2 | How should the executor preserve commit/abort/compensation semantics around a faulted scheduled step? | Calling `executeUntilStep` directly at a fault can bypass normal `Workflow.resume(...)`/abort handling and produce misleading results. | high | For a fault after a prefix, execute the non-faulting prefix, then call `resumeWorkflow`; for fault at first step or full-vector execution, use normal workflow lifecycle where possible. |
| Q3 | What runtime identity should the simulator fault provider query use? | Functionality simple name + step name is insufficient for same-functionality instances, multi-saga scenarios, and future parallel/search execution. | high | Use scenario execution attempt id, scenario plan id, saga instance id when bound, functionality FQN, scheduled step id when known, and runtime step name. |
| Q4 | Should CSV behavior be retained as an adapter/provider, and what compatibility must be preserved? | Existing Quizzes and simulator tests rely on behavior directories; breaking them broadens the feature unexpectedly. | medium | Keep CSV as a legacy/manual provider behind the same simulator abstraction, but do not use it from the scenario executor. |
| Q5 | What report fields make a fault execution reproducible enough for future scoring/search? | Later impact scoring and GA need duplicate avoidance and exact vector-to-slot evidence. | medium | Report assigned vector, default/explicit source, scenario execution attempt id, slot mapping, provider mode, faulted step if reached, and lifecycle terminal outcome. |
| Q6 | How should fault provider state be scoped and cleaned up? | Global/static impairment state can leak across tests or parallel executor runs. | medium | Use explicit scope/install/close semantics and report/validate cleanup in executor tests. |

## Notes

The selected approach is intentionally broader than the current single-saga executor implementation. The spec should avoid language that frames the work as a temporary POC, while still allowing dependency-ordered implementation slices.
