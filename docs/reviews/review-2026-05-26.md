**Date:** 2026-05-26

**Verdict:** Issues requiring action

> **Clean** = no boilerplate large enough to refactor; guidance is self-contained.
> **Minor issues** = repetition or drift present; no agent produces wrong output.
> **Issues requiring action** = repetition large enough to warrant a base class, or guidance gaps that cause silent coverage holes.

---

## Executive Summary

The `quizzes-full` test suite is correct, deterministic, and aligned with the doc taxonomy — 62 files cover T1, T2, Service-Command T2, and T3 with no false coverage gaps. The issues are not correctness; they are **scale-related boilerplate** (five `*InterInvariantTest` files repeat the same 6–10-line consumer-aggregate fixture chain ~30+ times in aggregate, and assertion-side `createUnitOfWork("check") + aggregateLoadAndRegisterRead` is written out four to six times per file) and **guidance fragmentation** that has begun to cost coverage. The interleaving-rule drift first flagged on 2026-05-23 (see `review-2026-05-23-deferred.md:11–27`) is the canonical example: seven concrete test classes are still missing scenarios because `session-b.md:153` only cross-references `testing.md:89–93` instead of restating the rule.

Part A is mostly a refactor opportunity (low risk, high readability win). Part B contains three Major items that should land before the next implement-aggregate session for a new app.

## Check A1 — Repetition: T3 fixture chain duplicated five times

Every `*InterInvariantTest` rebuilds the same upstream aggregate chain — Course → User → Execution → enroll → (Topic → Question → Quiz). The shape is identical and the differences are cosmetic (which `_2` user is created for the "ignores unrelated" branch, whether Quiz is needed).

| File | LOC | Setup style | Setup lines |
|------|----:|-------------|------------:|
| `ExecutionInterInvariantTest.groovy`   | 151 | Inline in every `given:`           | 4×4 lines = 16 |
| `QuestionInterInvariantTest.groovy`    | 112 | Inline in every `given:`           | ~6 per test    |
| `QuizInterInvariantTest.groovy`        | 162 | Inline in every `given:`           | ~8 per test    |
| `QuizAnswerInterInvariantTest.groovy`  | 322 | Shared `setup()` block (lines 33–53) | 16 (once)    |
| `TournamentInterInvariantTest.groovy`  | 356 | Shared `setup()` block             | ~24 (once)     |

Within a single file (`QuizAnswerInterInvariantTest.groovy:33–53`) the chain is hoisted into `setup()`, which is the right pattern. Across files the chain is rebuilt from scratch. There is no shared base class capturing it.

Severity: **Major** — not a bug, but the largest concentrated boilerplate in the suite. Refactor sketch in Check A6.

def checkUow = unitOfWorkService.createUnitOfWork("check")

def updated = loadForCheck(quizAnswer.aggregateId, QuizAnswer)

Severity: **Minor** — cosmetic but high frequency.

## Check A3 — Step-interleaving boilerplate

Every interleaving test re-instantiates the saga class directly:

```groovy
def uow1  = unitOfWorkService.createUnitOfWork("createExecution")
def func1 = new CreateExecutionFunctionalitySagas(
        unitOfWorkService, ACRONYM_1, ACADEMIC_TERM_1, courseDto.aggregateId,
        uow1, commandGateway)
func1.executeUntilStep("getCourseStep", uow1)
…
func1.resumeWorkflow(uow1)
```

The 5+-arg constructor mirrors `{Op}Functionalities` exactly, and the step-name string is duplicated as a magic literal. There are no constants for step names, so a typo (`"getCorseStep"`) compiles, runs, and never pauses the workflow — the test then asserts no exception was thrown and passes silently. This is the same pathology `review-tests/SKILL.md` Step 5.A line 210 already flags ("interleaving with no concurrent saga in between"), now reachable via typo as well as by structural mistake.

Severity: **Minor** for boilerplate; **Major** for the silent-pass typo risk. Add `public static final String STEP_GET_COURSE = "getCourseStep"` next to the step definition in `*FunctionalitySagas.java` and reference it from tests, or expose a `Steps` interface on each saga.

## Check A4 — Not-found magic numbers and Path A/B drift

`testing.md §Not-Found Assertions` (lines 226–259) splits not-found into Path A (`SimulatorException` from `aggregateLoadAndRegisterRead`) and Path B (`{App}Exception` from a custom-repository `Optional`-empty guard). Real tests apply both paths correctly, but:

- Not-found IDs are written as magic numbers (`999999`, `999`) per test rather than a named test constant. Pulling a `NONEXISTENT_AGGREGATE_ID = 999_999` into `QuizzesFullSpockTest` would remove ~20 occurrences and make the intent obvious in test bodies.
- `ExecutionInterInvariantTest.groovy:39–44` uses a non-standard Spock block sequence — `then:` immediately followed by `when:`/`then:` to capture the exception from a subsequent call. It works, but the established T3 deletion-event idiom (`testing.md:386–408`) is `and:` to extend the `when:` phase + `then: thrown(SimulatorException)`. Two patterns for the same intent across the same suite.

Severity: **Minor**.

## Check A5 — Phrasing drift between T3 files

Test method names express the same idea in different voices:

## Check A7 — Helper limitations on `QuizzesFullSpockTest`

The helpers in `QuizzesFullSpockTest.groovy:105–158` work but hard-code values that constrain what tests can express:

- `createQuestion` hard-codes a 2-option set with the first option correct (lines 123–129). No T2 test can vary correctness or option count without bypassing the helper.
- `createQuiz` hard-codes `availableDate = now + 1d`, `conclusionDate = now + 2d`, `resultsDate = now + 3d`, `type = "GENERATED"` (lines 131–140). Tests that need a started/concluded quiz can't use this helper and re-call `quizFunctionalities.createQuiz(...)` inline.
- `createExecution` constants `ACRONYM_1`, `ACADEMIC_TERM_1` are not defined on the base class — every caller passes literals. Adding constants would reduce magic strings.
- `createStartedTournament` is a useful overload pattern (lines 153–158); apply it to Quiz (`createStartedQuiz`) and reduce inline date arithmetic.

Severity: **Minor**.

## Check A9 — Refactor sketches

The two highest-value refactors are sketched below. They are illustrative, not API contracts.

### A9.1 — `InterInvariantTestBase`

```groovy
abstract class InterInvariantTestBase extends QuizzesFullSpockTest {

    Integer courseId, userId, executionId, topicId, questionId, quizId

    // Override in subclasses to pick which prerequisites to build.
    protected void buildFixture(Set<Stage> stages = [Stage.QUESTION] as Set) {
        courseId    = createCourse(COURSE_NAME_1, COURSE_TYPE_TECNICO).aggregateId
        userId      = createUser(USER_NAME_1, USER_USERNAME_1, STUDENT_ROLE).aggregateId
        executionId = createExecution(courseId, "EA001", "1st Semester 2024/25").aggregateId
        executionFunctionalities.enrollStudentInExecution(executionId, userId)
        if (Stage.QUESTION in stages || Stage.QUIZ in stages) {
            topicId    = createTopic(courseId, "Topic A").aggregateId
            questionId = createQuestion(courseId, [topicId], "Q1 Title", "Q1 Content").aggregateId
        }
        if (Stage.QUIZ in stages) {
            quizId = createQuiz(executionId, [questionId]).aggregateId
        }
    }

    enum Stage { ENROLLMENT, QUESTION, QUIZ }

    protected <T> T loadForCheck(Integer aggregateId, Class<T> type) {
        def uow = unitOfWorkService.createUnitOfWork("check")
        return type.cast(unitOfWorkService.aggregateLoadAndRegisterRead(aggregateId, uow))
    }
}
```

`QuizAnswerInterInvariantTest` then collapses to:

```groovy
def setup() { buildFixture([Stage.QUIZ] as Set) }

def "quizAnswer reflects UpdateStudentNameEvent"() {
    given:
    def quizAnswer = createQuizAnswer(quizId, userId)
    …
    then:
    loadForCheck(quizAnswer.aggregateId, QuizAnswer).userName == USER_NAME_2
}
```

Expected reduction: 30–40 LOC per T3 file × 5 files ≈ 150–200 LOC, with concentrated knowledge of "what a T3 fixture is" in one place.

In each `{Op}FunctionalitySagas.java`:

public static final class Steps {

# Part B — Guidance (testing.md + implement-aggregate skill)

## Check B1 — Major: T2 step-interleaving rule still produces silent coverage gaps

`session-b.md:153` says:

> Follow `docs/concepts/testing.md` § T2 — Step-interleaving rule (lines 89–93). **One interleaving case per saga step that calls `setSemanticLock` OR `setForbiddenStates` — no exceptions:**

The rule is in fact restated below it (lines 153–155 cover the two sub-rules). But the **per-step enumeration** that an agent has to perform — "for each `setSemanticLock`/`setForbiddenStates` call in the saga I just wrote, did I add an interleaving test?" — is not enforced anywhere. The result is documented in `review-2026-05-23-deferred.md:15–23`: seven concrete classes missing interleaving cases despite the rule being correct and the sagas being correct.

This is the canonical example of a rule that needs **mechanical verification**, not prose restatement. Options:

1. Add a checklist artifact to the session retro: "List every step in your sagas, mark each as `[locked]` / `[forbidden]` / `[plain]`, confirm a test exists for each." The retro then becomes a sentry against drift.
2. Add an `IT-` prefix grep convention to test names and require a count match between saga `setSemanticLock`/`setForbiddenStates` call sites and `IT-`-prefixed test methods per saga.
3. Build it into `review-tests/SKILL.md` Step 5 as an enumerable check rather than a smell.

Severity: **Major** — drift mechanism is known and recurring.

## Check B3 — Medium: Spec-table artifact has no defined output location

`testing.md:38–49` and `session-b.md:130–142` both mandate a spec table before writing assertions. Neither says **where it lives** — comment block in the test, retro entry, scratch buffer. In practice it lives in the agent's working memory and is not auditable.

Two observable consequences:

- A reviewer cannot tell whether the spec table was actually written or hallucinated from the implementation. The anti-pattern guard at `session-b.md:128` is unenforceable.
- When `plan.md` is later corrected, there is no record of which tests were written against the prior spec.

Recommended fix: require a `// Spec:` comment block at the head of each T2 method, three lines max:

```groovy
def "createExecution: success"() {
    // Spec: Execution.acronym = input; emits CreateExecutionEvent; SagaState = NOT_IN_SAGA.
    // Violations: NONE for happy path.
    // Source: plan.md §2.4 Execution / functionalities.
    …
}
```

Either that, or route the spec table into the session retro under a dedicated subsection so the artifact survives the session.

Severity: **Medium**.

## Check B4 — Medium: Named smells in `testing.md §Anti-Pattern` are incomplete relative to `review-tests`

`testing.md:23–29` lists five named smells and cross-references `review-tests/SKILL.md` Step 5.A for the full taxonomy. The cross-reference is accurate, but the testing.md list misses two recurring failure modes that `review-tests` catches and that occur in this codebase:

Adding two bullets to `testing.md:23–29` keeps the list self-contained.

## Check B5 — Medium: Two sources of truth for the anti-pattern checklist

`testing.md:19–31` and `review-tests/SKILL.md` Step 5.A both implement the Fake/Wrong/Weak detection checklist, with `review-tests` materially more detailed. Result:

- Agents writing tests read `testing.md` and miss the depth.
- Agents reviewing tests read `review-tests` and apply the full checklist.

The implementation pass is the one that should be most thorough; right now the review pass is. Either promote `review-tests` Step 5.A to authoritative and have `testing.md` quote a 3-bullet summary + link, or hoist the full checklist to `testing.md` and have `review-tests` reference it.

Severity: **Major** — agent behavior measurably differs by which doc was loaded.

Real tests universally use `setup()` (see `CourseCountsTest.groovy:23–25`, `QuizAnswerInterInvariantTest.groovy:33–53`). One sentence pinning the convention would prevent drift.

## Check B7 — Minor: Session "Reads" lists don't bound which sections to load

Each session file lists docs to read before producing code (`session-b.md:9–37`, `session-c.md:11–34`, `session-d.md:11–33`). The lists name **files** but not **line ranges or sub-sections**. For a long doc like `events.md` or `service.md`, this asks the agent to load the full file even when only one section applies. As a token-budget matter on a constrained run this can force the agent to truncate the most relevant section.

Adding line ranges or sub-section anchors (e.g., "`service.md` § Read-operation, lines 40–110") tightens the contract.

Severity: **Minor**.

## Check B8 — Minor: Session files overlap on saga-step content

Same applies to `session-d.md:128–138` ByEvent template vs `events.md:178` sagaState-guard rule.

Severity: **Minor**.
