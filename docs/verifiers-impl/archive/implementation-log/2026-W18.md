### Scenario catalog export

What changed:

- Added scenario model records for saga definitions, step definitions, step footprints, input variants, conflict evidence, scheduled steps, fault spaces, scenario plans, generation results, and manifests.
- Added bounded scenario generation with deterministic IDs.
- Added input normalization, input tuple joining, conflict graph building, connected saga-set enumeration, and schedule enumeration.
- Added JSONL catalog writer, rejected-input JSONL writer, and manifest writer.
- Accepted scenario JSONL includes input source-mode metadata.
- Rejected TCC/MIXED candidates are written to `scenario-catalog-rejected-inputs.jsonl`, even when empty.
- Stable catalog/manifest/rejected artifact filenames are written inside the verifier run output directory.
- Manifest output includes artifact paths, source-mode counters, and rejection counters.
- Added adapter from `ApplicationAnalysisState` into the scenario IR.
- Wired catalog export into `ScenarioGeneratorApplication` after HTML report generation.
- Added disabled-by-default configuration under `verifiers.scenario-catalog`, including `rejected-inputs-path`.
- Added `verifiers.output-root`, defaulting to `output` relative to the verifier working directory.
- Each run now writes to `<output-root>/<application>-<yyyyMMdd-HHmmss-SSS>/` instead of the analyzed application's tree or Maven `target`.
- The run directory is the archive; per-file timestamp sibling archives were removed and the manifest no longer exposes archive-path fields.
- Hardened relative configured output paths so they resolve under the verifier run output directory, cannot traverse outside it, and cannot cross existing symlink path segments; absolute artifact-path overrides are rejected.

Why it matters:

- The verifier now has a machine-readable output contract.
- Future execution should consume structured JSONL/manifest data instead of parsing the human HTML report.
- Static scenario synthesis is now represented as a concrete artifact rather than only as report prose.
- Catalog consumers can distinguish accepted saga inputs from rejected TCC/MIXED diagnostic candidates.

### Conservative adapter refinements

What changed:

- Saga dispatch footprints remain type-only unless exact key data exists.
- Missing aggregate names remain non-matchable/unknown rather than being mislabeled as type-only.
- Groovy constructor traces map to replayable, partial, or unresolved input variants based on recipe metadata and unresolved recipe kinds.

Why it matters:

- The catalog avoids overstating static certainty.
