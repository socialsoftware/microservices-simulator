## Completed

### Runtime functionality class identity

What changed:

- Dynamic evidence events can now carry:
  - `functionalityClassFqn`
  - `functionalityClassSimpleName`
- `ExecutionPlan` captures these values from the actual `WorkflowFunctionality` instance.
- `STEP_STARTED`, `STEP_FINISHED`, `COMMAND_SENT`, and `AGGREGATE_ACCESSED` inherit the class identity from the active dynamic evidence step context when present.
- Existing dynamic evidence APIs remain source-compatible for older callers that only provide `functionalityName`.

Why it matters:

- Runtime evidence no longer has to rely only on operational names such as `findQuiz`.
- The actual saga/functionality class can later be matched directly against static `InputVariant.sagaFqn`.

### Per-test dynamic input map

What changed:

- Added `DynamicInputMapWriter`.
- `DynamicEnrichmentOrchestrator` now writes `dynamic-input-map.json` into each per-class evidence directory before launching the Maven test class.
- Each map includes only accepted inputs present in final `ScenarioPlan.inputs()` whose `sourceClassFqn` matches the selected test class.
- Map entries include:
  - `inputVariantId`
  - `sagaFqn`
  - source class/method/binding metadata
  - source mode and resolution status
  - static step-name hints from the expanded schedule
  - simple literal argument hints such as `arg[0]: 42`
  - aggregate-type hints from conflict evidence when available
  - scenario plan ids
  - source/provenance diagnostics

Why it matters:

- The simulator has a small, per-test lookup table for runtime attribution.
- This keeps attribution generic and avoids adding application-specific hooks.

### Simulator-side input map loading and first-pass attribution

What changed:

- Added simulator-side `DynamicInputMap`, `DynamicInputMapLoader`, `DynamicInputAttribution`, and `DynamicInputAttributionHolder`.
- Added `simulator.dynamic-evidence.input-map-path`.
- `DynamicEnrichmentOrchestrator` now passes the per-class map path into each Maven test-class run.
- `DynamicEvidenceJsonlRecorder` loads the map when dynamic evidence is enabled and clears the holder when it closes.
- `DynamicEvidenceContext.enterStep(...)` resolves the active runtime step against the loaded map.
- `STEP_STARTED`, `STEP_FINISHED`, `COMMAND_SENT`, and `AGGREGATE_ACCESSED` propagate the resolved `inputVariantId` from the active step context.
- Step events also carry attribution diagnostics:
  - `MATCHED` with candidate id when exactly one input matches;
  - `NO_MATCH` when no candidate matches;
  - `AMBIGUOUS` with candidate ids when multiple static inputs match.

Why it matters:

- Runtime evidence can now say "this event belongs to input variant X" without changing Quizzes tests.
- The first-pass match is deliberately narrow: it uses test identity, actual functionality class FQN, and step name. It does not guess from simple names.
- Ambiguous cases are preserved as diagnostics for the next refinement slice instead of being converted into false exact matches.

### Quantification surface for dynamic improvement

What changed:

- `DynamicEvidenceJoinerSpec` now proves that direct `inputVariantId` evidence resolves an otherwise ambiguous static identity match at the joiner level.
- `DummyappDynamicEnrichmentIntegrationSpec` now writes two dynamic-evidence variants for the same selected dummyapp plan:
  - semantic-only evidence, which produces `MATCHED_HIGH_CONFIDENCE=1`;
  - the same evidence with direct runtime `inputVariantId`, which produces `MATCHED_EXACT=1`.
- The enriched manifest and join report remain the measurement surface. The relevant counts are `MATCHED_EXACT`, `MATCHED_HIGH_CONFIDENCE`, `AMBIGUOUS`, `UNMATCHED`, and `NOT_COVERED`.

Why it matters:

- The first-pass propagation should be read as a certainty upgrade, not automatically as more total matches.
- If a scenario was already high confidence, direct `inputVariantId` moves it to exact.
- If a scenario is ambiguous because several static inputs share the same test/functionality/step shape, this first-pass attribution will still report ambiguity. Reducing those cases needs command/aggregate/literal refinement.

### Quizzes before/after measurement

What changed:

- Installed the updated simulator artifact locally and reran the same sagas-only Quizzes selection used by the Task 8 baseline.
- The refreshed run selected the same `42` test classes, passed the same `40`, failed the same `2`, read the same `42` evidence files, and read the same `18868` dynamic events.
- Join counts changed from the old baseline:
  - before runtime `inputVariantId`: `MATCHED_EXACT=0`, `MATCHED_HIGH_CONFIDENCE=2`, `AMBIGUOUS=44`, `UNMATCHED=20`, `NOT_COVERED=0`, `warningCount=8238`;
  - after runtime `inputVariantId`: `MATCHED_EXACT=46`, `MATCHED_HIGH_CONFIDENCE=0`, `AMBIGUOUS=3`, `UNMATCHED=17`, `NOT_COVERED=0`, `warningCount=328`.
- Runtime attribution diagnostics show `226` step events with `payload.inputVariantAttributionStatus=MATCHED`, `270` command/aggregate events carrying the propagated top-level `inputVariantId`, and `8096` step events with `NO_MATCH`.

Why it matters:

- The main improvement is exactness: the verifier can now say which static input variant produced the runtime evidence for most covered Quizzes plans.
- Total covered/non-unmatched plans moved only from `46` to `49`, so this should not be oversold as "dynamic covers everything now".
- The remaining `AMBIGUOUS=3` cases are same-shape neighboring-input cases where direct runtime ids exist, but point to other static input variants than the plan being enriched.
- The remaining `UNMATCHED=17` records need separate review before claiming the tests exercise every useful static input.
