# VSCode Extension

This chapter explains the architecture of the Nebula VSCode extension, how to build and package it, and how to release new versions.

## Overview

The Nebula VSCode extension provides language support for `.nebula` files:

- Syntax highlighting via TextMate grammar
- Error detection and diagnostics via the Langium language server
- Code completion and navigation
- Bracket matching and comment toggling

The extension follows the standard Language Server Protocol (LSP) architecture: a **language client** runs inside VSCode, communicating over IPC with a **language server** that performs parsing, validation, and analysis.

```
┌─────────────────────────────────────────────────┐
│                    VSCode                        │
│                                                  │
│  ┌─────────────┐          ┌──────────────────┐  │
│  │  Extension   │   IPC    │  Language Server  │  │
│  │  (Client)    │◄────────►│  (Langium)       │  │
│  │  main.ts     │          │  main.ts         │  │
│  └─────────────┘          └──────────────────┘  │
│         │                          │             │
│         ▼                          ▼             │
│  TextMate Grammar           Grammar Parser       │
│  (syntax colors)         (diagnostics, LSP)      │
└─────────────────────────────────────────────────┘
```

## Extension Architecture

### Key Files

```
dsl/nebula/
├── src/
│   ├── extension/
│   │   └── main.ts              # VSCode extension entry point (client)
│   └── language/
│       ├── main.ts              # Language server entry point
│       ├── nebula-module.ts     # Service registration and DI
│       ├── nebula.langium       # Grammar definition
│       ├── validation/          # Semantic validators
│       └── generated/           # Langium-generated parser
├── syntaxes/
│   └── nebula.tmLanguage.json   # TextMate grammar (syntax highlighting)
├── package.json                 # Extension manifest and configuration
├── langium-config.json          # Langium project configuration
├── esbuild.mjs                  # Bundler configuration
└── release.sh                   # Release automation script
```

### Extension Client (`src/extension/main.ts`)

The extension client manages the lifecycle of the language server:

```typescript
// Activation: starts the language server when a .nebula file is opened
export function activate(context: vscode.ExtensionContext): void {
    client = startLanguageClient(context);
}

// Deactivation: stops the language server when the extension is disabled
export function deactivate(): Thenable<void> | undefined {
    if (client) {
        return client.stop();
    }
    return undefined;
}
```

The client is configured to:
- Load the server module from `out/language/main.cjs`
- Communicate via IPC transport
- Activate on `.nebula` files (`documentSelector: [{ scheme: 'file', language: 'nebula' }]`)

### Language Server (`src/language/main.ts`)

The server uses Langium's built-in LSP implementation:

```typescript
const connection = createConnection(ProposedFeatures.all);
const { shared } = createNebulaServices({ connection, ...NodeFileSystem });
startLanguageServer(shared);
```

This provides all LSP features (diagnostics, completion, hover, go-to-definition) based on the Langium grammar and registered validators.

### Service Module (`src/language/nebula-module.ts`)

The module registers custom services into Langium's dependency injection:

```typescript
export const NebulaModule = {
    validation: {
        NebulaValidator: (services) => new NebulaValidator(services),
    },
    references: {
        ScopeComputation: (services) => new NebulaScopeComputation(services),
        ScopeProvider: (services) => new NebulaScopeProvider(services),
    },
};
```

- **NebulaValidator** -- semantic validation rules (e.g., root entity required, valid cross-aggregate references)
- **NebulaScopeComputation** -- determines which symbols are exported from each file
- **NebulaScopeProvider** -- resolves cross-file references (e.g., `from Category` finds the Category aggregate)

### Extension Manifest (`package.json`)

The `contributes` section registers the language with VSCode:

```json
{
    "contributes": {
        "languages": [{
            "id": "nebula",
            "aliases": ["Nebula", "nebula"],
            "extensions": [".nebula"]
        }],
        "grammars": [{
            "language": "nebula",
            "scopeName": "source.nebula",
            "path": "syntaxes/nebula.tmLanguage.json"
        }]
    },
    "activationEvents": ["onLanguage:nebula"]
}
```

### TextMate Grammar (`syntaxes/nebula.tmLanguage.json`)

Provides syntax highlighting by matching token patterns:

- **Keywords** -- `Aggregate`, `Entity`, `Events`, `Repository`, `Service`, `WebAPIEndpoints`, `@GenerateCrud`, `@Query`, etc.
- **Strings** -- double-quoted strings with escape sequences
- **Comments** -- line comments (`//`) and block comments (`/* */`)

This file is auto-generated by Langium from the grammar definition. To regenerate:

```bash
npm run langium:generate
```

## Building the Extension

### Prerequisites

```bash
node --version    # v18.0.0 or higher
npm --version     # 9.0.0 or higher
code --version    # VSCode 1.67.0 or higher
```

### Build Steps

The extension uses a two-stage build:

1. **TypeScript compilation** (`tsc`) -- type-checks the source and produces `.js` files in `out/`
2. **esbuild bundling** (`esbuild.mjs`) -- bundles the extension client and language server into single `.cjs` files

```bash
cd dsl/nebula

# Install dependencies
npm install

# Generate Langium parser (required after grammar changes)
npm run langium:generate

# Build everything
npm run build
```

The build produces:
- `out/extension/main.cjs` -- bundled extension client
- `out/language/main.cjs` -- bundled language server
- `out/cli/` -- CLI and generator code

### esbuild Configuration

The bundler (`esbuild.mjs`) is configured for VSCode's extension host requirements:

| Setting | Value | Reason |
|---------|-------|--------|
| `format` | `cjs` | VSCode extension host requires CommonJS |
| `outExtension` | `.cjs` | Prevents Node.js ESM/CJS confusion |
| `external` | `['vscode']` | VSCode API is provided at runtime |
| `platform` | `node` | Extensions run in Node.js |
| `target` | `ES2017` | Compatibility with VSCode's runtime |

### Watch Mode

For development, use watch mode to auto-rebuild on changes:

```bash
# Watch both TypeScript and esbuild
npm run watch

# Watch only grammar changes
npm run langium:watch
```

## Packaging the Extension

Extensions are distributed as `.vsix` files (ZIP archives with a specific structure).

### Using the Release Script

The project includes `release.sh` to automate packaging:

```bash
cd dsl/nebula

# Patch release (0.0.2 -> 0.0.3)
npm run release

# Minor release (0.0.2 -> 0.1.0)
npm run release:minor

# Major release (0.0.2 -> 1.0.0)
npm run release:major
```

The script performs the following steps:

1. Reads the current version from `package.json`
2. Bumps the version (major, minor, or patch)
3. Updates `package.json` with the new version
4. Runs `npm run langium:generate` to regenerate the parser
5. Runs `npm run build` to compile and bundle
6. Packages the extension using `@vscode/vsce`
7. Outputs the `.vsix` file to `dsl/extensions/`

Output:
```
Current version: 0.0.2

New version: 0.0.3

Updated package.json

Building...
...

Packaging extension...
...
Extension packaged: dsl/extensions/nebula-extension-0.0.3.vsix

To install:
  code --install-extension dsl/extensions/nebula-extension-0.0.3.vsix
```

### Manual Packaging

To package without bumping the version:

```bash
cd dsl/nebula

# Build first
npm run langium:generate
npm run build

# Package
npx @vscode/vsce package --no-dependencies -o ../extensions/nebula-extension-manual.vsix
```

The `--no-dependencies` flag skips bundling `node_modules` into the `.vsix` since esbuild already bundles everything.

## Langium Configuration

The `langium-config.json` file configures the Langium framework:

```json
{
    "projectName": "Nebula",
    "languages": [{
        "id": "nebula",
        "grammar": "src/language/nebula.langium",
        "fileExtensions": [".nebula"],
        "textMate": {
            "out": "syntaxes/nebula.tmLanguage.json"
        }
    }],
    "out": "src/language/generated"
}
```

- **grammar** -- path to the Langium grammar file
- **fileExtensions** -- file extensions recognized by the language server
- **textMate.out** -- output path for the auto-generated TextMate grammar
- **out** -- output directory for the generated parser, AST types, and module

When you run `npm run langium:generate`, Langium reads this configuration and produces:
- `src/language/generated/ast.ts` -- TypeScript interfaces for the AST
- `src/language/generated/grammar.ts` -- parser implementation
- `src/language/generated/module.ts` -- generated service module
- `syntaxes/nebula.tmLanguage.json` -- TextMate grammar for syntax highlighting

## Modifying the Extension

### Adding a New Keyword to Syntax Highlighting

1. Add the keyword to the grammar (`src/language/nebula.langium`)
2. Regenerate: `npm run langium:generate`
3. The TextMate grammar at `syntaxes/nebula.tmLanguage.json` is updated automatically
4. Rebuild and repackage: `npm run build`

### Adding a New Validator

1. Add validation logic in `src/language/validation/`
2. Register the check in the validator's `registerValidationChecks()` method
3. Rebuild: `npm run build`
4. The language server will now report new diagnostics in VSCode

### Debugging the Extension

1. Open the `dsl/nebula/` folder in VSCode
2. Press `F5` to launch a new Extension Development Host window
3. Open a `.nebula` file in the new window
4. The extension runs with debugger attached
5. Set breakpoints in `src/extension/main.ts` or `src/language/main.ts`

For language server debugging, set the `DEBUG_SOCKET` environment variable:

```bash
DEBUG_SOCKET=6009 code --extensionDevelopmentPath=./dsl/nebula
```

## Release Checklist

When preparing a new extension release:

1. Ensure all grammar changes are finalized
2. Run `npm run langium:generate` to regenerate the parser
3. Run `npm run build` to verify the build succeeds
4. Test the extension locally (F5 in VSCode)
5. Run `npm run release` (or `release:minor`/`release:major`)
6. Install and verify the packaged `.vsix`
7. The `.vsix` file is output to `dsl/extensions/`
